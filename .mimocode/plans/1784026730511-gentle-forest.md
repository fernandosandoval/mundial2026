# Plan: Mejoras robustas al Monitor Mundial 2026

## Resumen

Cinco mejoras alineadas: caché + rate limiter, aprovechamiento completo de la API, persistencia de jobs, emails HTML en todas las notificaciones, y live tracking durante los partidos. Todo manteniendo los patrones existentes (DI, interfaces, test doubles).

---

## 1. Cache + Rate Limiter + Retry

**Objetivo:** No repetir llamadas a la API, respetar el límite de 10 req/min, y recuperarse de errores transitorios.

### Archivos a modificar/crear
- **Nuevo:** `src/apiCache.ts` — caché genérico con TTL
- **Nuevo:** `src/retry.ts` — retry con exponential backoff
- **Modificar:** `src/apiClient.ts` — integrar caché y retry

### Diseño

**`apiCache.ts`** — Caché in-memory con TTL:
```ts
class ApiCache<T> {
  private store = new Map<string, { data: T; expiresAt: number }>();
  
  get(key: string): T | null // retorna null si expiró o no existe
  set(key: string, data: T, ttlMs: number): void
  clear(): void
}
```

**`retry.ts`** — Retry con backoff exponencial:
```ts
async function withRetry<T>(
  fn: () => Promise<T>,
  options?: { maxRetries?: number; baseDelayMs?: number; maxDelayMs?: number }
): Promise<T>
```
- Default: 3 reintentos, delay base 1000ms, max 10000ms
- Maneja 429 (rate limit) con el header `Retry-After` si está disponible
- Maneja errores de red (ECONNRESET, ETIMEDOUT)

**`apiClient.ts`** — Cambios:
- `fetchCompetitionMatches()` usa caché con TTL de 60 segundos (la API rate limit es 10/min, un缓存 de 60s asegura máx 1 req/min para esta ruta)
- Todas las llamadas HTTP pasan por `withRetry()`
- `getMatchById()` también se cachea (TTL 30s para live tracking)
- Agregar `clearCache()` para forzar refresh cuando se necesita

### Tests
- `tests/apiCache.spec.ts` — TTL, expiración, clear
- `tests/retry.spec.ts` — retry en error, límite de reintentos, manejo de 429

## 3. Persistencia de jobs y recuperación en restart

**Objetivo:** Que los notificaciones sobrevivan reinicios del proceso.

### Archivos a modificar/crear
- **Nuevo:** `src/jobStore.ts` — almacenamiento persistente de jobs
- **Modificar:** `src/scheduler.ts` — usar jobStore y reconciliar en startup
- **Modificar:** `src/index.ts` — pasar jobStore al scheduler

### Diseño

**Opción elegida: JSON file** (simple, sin dependencias externas, suficiente para un app de un Mundial)

```ts
interface PersistedJob {
  id: string;            // e.g. "match-1001-hour-before"
  matchId: number;
  type: 'one-hour-before' | 'kickoff' | 'match-end-check' | 'live-tracking';
  scheduledAt: string;   // ISO date
  status: 'pending' | 'sent' | 'missed';
  createdAt: string;
}

class JobStore {
  private filePath: string;
  
  save(job: PersistedJob): void
  markSent(jobId: string): void
  markMissed(jobId: string): void
  getPendingJobs(): PersistedJob[]
  getMissedJobs(): PersistedJob[]
  cleanup(olderThanMs: number): void
}
```

**Archivo de persistencia:** `data/jobs.json` (carpeta `data/` en .gitignore)

**Reconciliación en startup (`scheduler.ts`):**
1. Al `initialize()`, cargar jobs pendientes del store
2. Para cada job pendiente cuya fecha ya pasó → marcar como `missed`
3. Para cada job `missed` tipo `match-end-check` → ejecutar `checkMatchResultAndNotify()` inmediatamente
4. Para cada job `missed` tipo `one-hour-before` o `kickoff` → si el partido sigue en curso o no ha pasado mucho, enviar la notificación de todas formas
5. Los jobs nuevos se guardan en el store al programarlos
6. Al enviar un email, marcar el job como `sent`
7. Cleanup periódico: borrar jobs `sent` de hace >7 días

## 5. Live Tracking durante el partido

**Objetivo:** Notificaciones en tiempo real durante el partido: mitad, goles, fin.

### Archivos a modificar/crear
- **Nuevo:** `src/liveTracker.ts` — polling loop durante partidos en vivo
- **Modificar:** `src/scheduler.ts` — integrar liveTracker
- **Modificar:** `src/types.ts` — tipos para live tracking
- **Modificar:** `src/config.ts` — intervalo de polling configurable

### Diseño

**`liveTracker.ts`**:
```ts
interface LiveTrackerOptions {
  apiClient: FootballDataApi;
  emailService: EmailService;
  matchId: number;
  matchInfo: MatchInfo;
  pollIntervalMs?: number;  // default 60_000 (1 min)
}

class LiveMatchTracker {
  private intervalId: NodeJS.Timeout | null = null;
  private lastGoalCount = 0;
  private lastStatus: MatchStatus;
  private halftimeEmailSent = false;

  start(): void    // inicia polling
  stop(): void     // detiene polling
  
  private async poll(): Promise<void>
  // 1. fetch match by ID
  // 2. comparar status anterior vs actual
  // 3. si status cambió a PAUSED → enviar halftime email
  // 4. siGoals.length > lastGoalCount → enviar goal email
  // 5. si status cambió a FINISHED → enviar result email y stop()
}
```

**Integración en `scheduler.ts`:**
- Cuando el status de un partido es `IN_PLAY` o el tiempo de kickoff llega, iniciar el tracker
- El tracker reemplaza la lógica de "estimated end check" para partidos en curso
- Si el proceso se reinicia durante un partido, el startup reconciliation detecta el match como `IN_PLAY` y reinicia el tracker

**Configuración:**
```ts
liveTrackingPollIntervalMs: Number(optionalEnv('LIVE_POLL_INTERVAL_MS', '60000'))
```

### Emails del live tracker
1. **Gol:** "¡Gol de [equipo]! [scorer] al [minuto]'. [marcador actual]"
2. **Descanso:** "Descanso: [home] [x] - [y] [away]. [resumen]"
3. **Resultado final:** Email completo con estadísticas
4. **Tarjeta roja:** "🔴 [player] expulsado ([team])"

- `tests/liveTracker.spec.ts` — cambios de status, detección de goles, detección de halftime
